# 🏗️ Structural Mechanics AI Agent

Автономний ШІ-агент **інженера-конструктора**, який розв'язує задачі
будівельної механіки — статично визначені **балки** (у т.ч. багатопрогінні
шарнірні балки Гербера), **ферми** (у т.ч. шпренгельні), **тришарнірні
арки** та **статично визначені рами** — і для кожної задачі:

1. 🧮 виконує повний аналітичний розрахунок (реакції опор, епюри M/Q/N,
   лінії впливу, переміщення за формулою Мора);
2. 🖥️ генерує інтерактивний HTML/JS-симулятор із повзунками навантажень
   і геометрії та canvas-візуалізацією схеми й епюр;
3. 📄 формує стандартизований звіт **.docx** із розрахунковою схемою,
   епюрами-рисунками, покроковим розв'язком і зведеними таблицями —
   оформлений за єдиним, наперед заданим шаблоном;
4. 🎓 **самонавчається** — якщо задача не схожа на жоден із наявних
   прикладів, агент пише новий симулятор із нуля і, за бажанням
   користувача, зберігає його у власній базі знань для використання
   в майбутніх сесіях;
5. 🔁 **самоперевіряється й самовиправляється** — перш ніж показати
   результат, агент перевіряє контрольні суми рівноваги (ΣFx, ΣFy, ΣM);
   якщо вони не близькі до нуля, він автоматично перераховує задачу
   ще раз, не чекаючи, поки помилку знайде людина.

Агент повністю працює на **[Ollama](https://ollama.com)** — локальній
LLM-моделі на вашому комп'ютері чи Codespace. Жодних викликів до
хмарних API (OpenAI, Anthropic тощо) не робиться — ваші розрахункові
дані нікуди не йдуть за межі вашої машини.

## 📐 Що саме вміє агент

| Тип конструкції | Що рахує | Word-шаблон |
|---|---|---|
| Багатопрогінна шарнірна балка (балка Гербера) | Поверхова схема, опорні реакції, епюри M/Q, лінії впливу | `report_beam.docx` |
| Шпренгельна ферма | Опорні реакції, зусилля в стрижнях (метод перерізів + вирізання вузлів) | `report_truss.docx` |
| Тришарнірна арка | Реакції, розпір, зусилля M/Q/N у перерізах, лінії впливу | `report_arch.docx` |
| Статично визначена рама | Реакції, епюри M/Q/N, переміщення за Мором–Верещагіним | `report_frame.docx` |

Якщо ви поставите задачу з іншою, невідомою агенту схемою (наприклад,
трипрогінна консольна балка з іншою розстановкою шарнірів, або ферма
іншого обрису) — агент напише новий симулятор за аналогією й попросить
вас підтвердити, чи зберегти його в базу знань під новим типом.

## 🧠 Як влаштований агент (архітектура)

```
Запит користувача
      │
      ▼
knowledge_base.py  ──► шукає 1-2 найрелевантніші приклади симуляторів
      │                 (RAG за ключовими словами, а не "злив" усієї бази)
      ▼
prompts.py  ──► збирає системний промпт: інженерні правила ДСТУ +
      │          точні JSON-схеми під кожен тип звіту + приклади коду
      ▼
Ollama (локальна LLM)  ──► текстовий розв'язок + HTML-код + JSON
      │
      ▼
verification.py  ──► перевіряє ΣFx/ΣFy/ΣM ≈ 0
      │                 ні → повертає задачу назад у Ollama з зауваженням
      │                 (до 2 самокорекцій), так → приймає результат
      ▼
Streamlit UI  ──► показує інтерактивний симулятор
      │
      ├─► (опційно) capture_js.py знімає PNG зі схеми/епюр canvas
      │
      └─► report_export.py збирає фінальний .docx (docxtpl + InlineImage)
```


## 🚀 Запуск на GitHub Codespaces (найпростіший спосіб)

1. Відкрийте репозиторій → **Code → Codespaces → Create codespace on main**.
2. У терміналі Codespace встановіть Ollama:
   ```bash
   curl -fsSL https://ollama.com/install.sh | sh
   ```
3. Запустіть Ollama-сервер у фоні:
   ```bash
   ollama serve &
   ```
4. Завантажте модель (рекомендовано — див. таблицю нижче):
   ```bash
   ollama pull qwen2.5-coder:7b
   ```
5. Встановіть Python-залежності:
   ```bash
   python -m pip install -r requirements.txt
   ```
6. Запустіть застосунок:
   ```bash
   python -m streamlit run app.py
   ```
7. Codespaces запропонує відкрити порт `8501` у браузері — погодьтесь,
   або відкрийте URL із виводу терміналу (напр. `http://localhost:8501`).

## 🤖 Вибір моделі Ollama

Модель обирається прямо в інтерфейсі (бічна панель → "Модель генерації").
Орієнтовні рекомендації за потужністю машини:

| Модель | VRAM/RAM | Якість розрахунків | Коли обирати |
|---|---|---|---|
| `qwen2.5-coder:1.5b` | ~2 ГБ | базова, більше помилок рівноваги | слабке залізо / лише спробувати |
| `qwen2.5-coder:7b` | ~8 ГБ | **рекомендовано за замовчуванням** | звичайний ноутбук / Codespace |
| `qwen2.5-coder:14b` | ~16 ГБ | найточніші розрахунки | потужна машина / GPU |
| `deepseek-coder-v2:16b` | ~16 ГБ | сильна альтернатива для складних рам/арок | GPU з 16+ ГБ VRAM |

Чим більша модель — тим менше циклів самокорекції знадобиться (агент
все одно перевірить рівновагу та перерахує сам, але краще, коли
помилок менше з самого початку).

## 🖱️ Як користуватись

1. Опишіть задачу в чаті природною мовою, наприклад:
   > Розрахувати тришарнірну арку прольотом 24 м, стрілою підйому 6 м,
   > розподілене навантаження q = 18 кН/м, зосереджена сила P = 40 кН
   > на відстані 8 м від лівої опори.
2. Агент виведе хід розв'язку в чат і відкриє інтерактивний симулятор
   зверху сторінки — можна покрутити повзунки, щоб побачити, як
   змінюються епюри при інших даних.
3. Якщо агент сам помітить, що ΣFy чи ΣM суттєво відрізняється від
   нуля — він покаже жовте попередження "🔁 Агент виявив похибку…" і
   перерахує задачу без вашого втручання (до 2 спроб).
4. У бічній панелі:
   - **📸 Зняти зображення з симулятора** — робить PNG-знімки поточної
     розрахункової схеми та епюри прямо з canvas симулятора.
   - **📥 Згенерувати .docx звіт** — збирає фінальний Word-документ за
     стандартизованим шаблоном, підставляючи розраховані числа, таблиці
     та зняті зображення.
   - **💾 Запам'ятати у базі знань** — якщо симулятор новий і вдалий,
     збережіть його: наступного разу агент використає його як приклад
     стилю для схожих задач (це і є "самонавчання").

## 🖼️ Про вставку зображень у Word (важливо)

Симулятор рендериться у власному `<iframe>` (через `st.components.v1.html`).
Кнопка «📸 Зняти зображення з симулятора» виконує невеликий JavaScript
(через пакет `streamlit-javascript`) у контексті головної сторінки,
знаходить потрібний `<canvas>` усередині iframe і знімає з нього
`toDataURL('image/png')`. Це працює "з коробки" для вбудованих
шаблонів (`beam_gerber.html`, `truss_sprengel.html`, `arch_base.html`,
`frame_base.html`) та для будь-якого нового симулятора, який агент
згенерує, дотримуючись контракту з `prompts.py` (canvas з
`id="mainCanvas"`, кнопка головної епюри з `id="btnDiagM"` тощо).

**Обмеження:** для вбудованих (легасі) шаблонів окремого canvas для
ліній впливу/одиничного навантаження немає — в звіті ці поля
лишаться порожніми, доки ви не попросите агента переробити симулятор
з окремими `influenceCanvas` / `unitLoadCanvas`, або не збережете
новий варіант через самонавчання.

Якщо кнопка не знаходить canvas (рідко — залежить від того, як
Streamlit версіонує iframe), просто натисніть її ще раз після того,
як симулятор повністю відрендерився.

## 🔁 Як працює самоперевірка й самокорекція

Кожен тип конструкції має свій набір контрольних сум рівноваги
(`check_Y`, `check_X`, `check_M` — залежно від типу), які модель
зобов'язана порахувати сама й повернути в JSON. Python-код у
`agent/verification.py` парсить ці значення і порівнює з нулем у
межах допуску (±0.5). Якщо десь похибка більша — агент автоматично
надсилає моделі повідомлення на кшталт:

> Самоперевірка виявила помилку рівноваги: перевірка «check_Y» = 3.42,
> що не близько до нуля. Перерахуй задачу заново…

і повторює цикл (максимум `MAX_SELF_CORRECTIONS = 2` рази), перш ніж
показати фінальний результат користувачу.

## 🎓 Як працює самонавчання

Замість того, щоб при кожному запиті вантажити в контекст LLM *весь*
вміст усіх файлів `simulators/templates/*.html` (це швидко переповнює
контекст, особливо в малих моделей), агент веде легкий індекс
`simulators/templates/index.json` з коротким описом і ключовими
словами кожного шаблону. У запит підмішується лише 1-2 найрелевантніші
приклади (`agent/knowledge_base.py::get_relevant_templates`).

Коли ви натискаєте «Запам'ятати у базі знань», новий `.html`-файл
зберігається на диск, а запис про нього (назва, тип, ключові слова)
дописується в `index.json` — з наступного запиту агент вже підбирає
цей приклад як релевантний, якщо схожа задача повториться.

> ⚠️ У Streamlit Cloud / Codespaces файлова система, як правило,
> ефемерна — після перезапуску контейнера вивчені шаблони можуть
> зникнути, якщо ви не закомітили їх у git або не підключили
> постійний диск.

## 🛠️ Розширення новими типами конструкцій

1. Додайте новий тип у `agent/schemas.py` (словник `SCHEMAS`) — опишіть
   точні ключі JSON, які відповідають вашому новому Word-шаблону.
2. Створіть `simulators/templates/report_<новий_тип>.docx` з Jinja-
   плейсхолдерами `{{ поле }}` та `{% for %}`-циклами під ті самі ключі
   (докладніше про синтаксис — у документації
   [docxtpl](https://docxtpl.readthedocs.io/)).
3. Додайте синоніми типу в `SYNONYMS` у `agent/knowledge_base.py`, щоб
   агент розпізнавав запити про новий тип конструкції.
4. Розв'яжіть кілька прикладних задач у чаті — щойно результат вас
   влаштує, збережіть симулятор через «Запам'ятати у базі знань», і
   він стане першим прикладом для нового типу.

## ❓ Усунення несправностей

| Проблема | Рішення |
|---|---|
| `Помилка сервера Ollama: ...` | Переконайтесь, що `ollama serve` запущено в іншому терміналі/фоні. |
| Модель довго "думає" або обрізає код | Зменшіть складність запиту або оберіть потужнішу модель (14b); перевірте `num_predict` у `app.py`. |
| `simulator_not_found` при знятті зображень | Дочекайтесь повного рендеру симулятора (кілька секунд) і натисніть кнопку ще раз. |
| У звіті порожні рисунки епюр | Спершу натисніть «📸 Зняти зображення з симулятора», лише потім «📥 Згенерувати .docx». |
| `Шаблон ... не знайдено!` при експорті | Перевірте, що поле `type` у JSON-відповіді моделі відповідає одному з `beam/truss/arch/frame`. |

## 📜 Ліцензія

Проєкт поширюється "як є", без гарантій точності інженерних розрахунків.
**Завжди перевіряйте результати кваліфікованим інженером перед
використанням у реальному проєктуванні** — це навчально-дослідницький
інструмент, а не сертифікований засіб проєктування.
